## 7. Investigation: client-beacon wiring

**Finding: not broken — never wired.** All three client-fired event kinds have **zero call sites** in the codebase.

Evidence:
- `track("signup_clicked", ...)` — referenced only in the JSDoc example at `src/lib/track.ts:7`. No real call sites.
- `track("checkout_clicked", ...)` — zero call sites.
- `track("email_report_opened", ...)` — zero call sites.
- `import ... from "@/lib/track"` — zero importers (the file is a dead module today).

This perfectly explains the 28d data: server-side events (`scan_started` 81, `scan_completed` 76, `scan_rate_limited` 2, `email_report_sent` 1) all log normally because `lib/analytics.ts:logEvent` is called from API routes. Client funnel kinds are all 0 because no UI fires them.

### Surfaces that should fire `signup_clicked`

| File | Line | What it is |
|---|---|---|
| `src/app/page.tsx` | 184 | Top-nav "Sign In" link → `/login` |
| `src/app/page.tsx` | 308 | Free-scan-limit error CTA "Create a Free Account" → `/login` |
| `src/components/ScanReport.tsx` | 156 | Scan-result card "Create Free Account" → `/login` |
| `src/components/ScanReport.tsx` | 398 | Inline "create a free account" link → `/login` |

All four are bare `<a href="/login">` with no `onClick`. They navigate before any handler gets a chance to fire. (Note: the prominent top-nav "Dashboard" button at `page.tsx:191` also redirects unauthenticated users to `/login` via the dashboard layout — same problem.)

### Surface that should fire `checkout_clicked`

- `src/app/dashboard/billing/CheckoutButton.tsx` — the `handleCheckout` function calls `/api/checkout` directly, no `track()`.
- `src/app/page.tsx:314` — the homepage `handleCheckout("pro")` button on the limit-hit error state, also no `track()`.

### Surface that should fire `email_report_opened`

This kind only makes sense if the email template embeds a 1×1 pixel or tracker URL pointing at a route that calls `track()` server-side. No such route exists. The kind is probably aspirational and can be dropped from the allowlist until an email-tracking strategy is decided.

### Recommended fix (not implemented)

Smallest-surface change:

1. Convert the four `/login` `<a>` tags to a small client component (`<SignupLink from="...">`) that calls `track("signup_clicked", { from })` in `onMouseDown` (fires on left, middle, ctrl-click — survives navigation via `sendBeacon`), then lets the default navigation proceed. **Do not** use `onClick` + `preventDefault` + `window.location` — it breaks middle-click and ctrl-click.
2. Add `track("checkout_clicked", { plan })` at the top of `handleCheckout` in both `CheckoutButton.tsx` and `page.tsx`.
3. Drop `email_report_opened` from `ALLOWED_KINDS` in `src/app/api/events/route.ts` (and the `ClientEventKind` union) until there's a corresponding sender. A truly aspirational kind in the allowlist is just clutter.

Effort: ~1 small PR, <30 lines diff, no schema change. Ships measurement of the most-load-bearing funnel step (`signup_clicked / scan_completed`) which is currently a black hole.
